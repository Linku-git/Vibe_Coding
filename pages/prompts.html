Perform a mid-project architecture review:

1. PRD Alignment Check
   - Which PRD requirements are fully implemented?
   - Which are partially done? What's missing?
   - Any features that drifted from the PRD?

2. Architecture Health
   - Are there any circular dependencies?
   - Is the file structure still clean and consistent?
   - Are naming conventions followed everywhere?
   - Any code that should be refactored before continuing?

3. Test Coverage
   - What percentage of routes/components have tests?
   - Which critical paths are untested?
   - Are there any flaky or skipped tests?

4. Documentation Freshness
   - Is progress.md accurate?
   - Are api-routes.md and frontend-pages.md up to date?
   - Any stale or contradictory documentation?

5. Technical Debt
   - List all TODOs and FIXMEs in the codebase
   - Identify hardcoded values that should be config
   - Flag any security concerns

6. Recommendations
   - What should be fixed before continuing?
   - Should any remaining sessions be re-scoped?
   - Priority order for fixes

Output as a structured report. Do NOT make changes — report only.
Read design/DESIGN.md for the complete design system.

Integrate the design tokens into the codebase:

1. Design Tokens
   - Extract all color tokens and create CSS variables / theme config
   - Extract typography tokens (font families, sizes, weights, line heights)
   - Extract spacing tokens (padding, margin, gap values)
   - Extract border radius, shadow, and animation tokens

2. Component Mapping
   - For each component in design/components/:
     - Create a matching frontend component
     - Apply exact design tokens (no hardcoded values)
     - Include all states: default, hover, active, disabled, loading
     - Ensure responsive behavior matches design

3. Screen Implementation
   - For each screen in design/screens/:
     - Build the page layout matching the design exactly
     - Use the component library created above
     - Match spacing, alignment, and visual hierarchy

4. Verification
   - Compare each implemented screen to the design screenshot
   - List any deviations with justification
   - Run visual regression tests if available

Do not invent colors, spacing, or layouts not in DESIGN.md.
If something is ambiguous, flag it rather than guessing.

Prepare the project for production deployment:

1. Environment Configuration
   - Verify all secrets use environment variables (no hardcoded values)
   - Create .env.example with all required variables documented
   - Ensure dev/staging/production configs are separated

2. Build Optimization
   - Run production build and fix any build errors
   - Check bundle size and identify optimization opportunities
   - Ensure all assets are optimized (images, fonts)
   - Verify tree-shaking is working

3. Security Checklist
   - HTTPS enforced
   - CORS configured correctly
   - Rate limiting on API endpoints
   - Input validation on all user inputs
   - SQL injection protection verified
   - XSS protection headers set
   - Authentication tokens properly secured
   - Sensitive data not logged

4. CI/CD Setup
   - Create GitHub Actions workflow (or equivalent)
   - Run linter, type checker, and tests on every PR
   - Auto-deploy to staging on merge to main
   - Manual promotion to production

5. Monitoring
   - Error tracking setup (Sentry or equivalent)
   - Health check endpoint: GET /api/health
   - Key metrics to monitor: response time, error rate, uptime

6. Documentation
   - Update Docs/readme.md with deployment instructions
   - Document rollback procedure
   - List all external service dependencies
