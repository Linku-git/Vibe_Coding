The Five Pillars

Agents define behavior. Skills encode procedures. MCP Servers connect to external tools. Tools are the agent's hands. Memory is the persistent brain. Together, they turn an AI from a chatbot into a reliable development partner.

6.1 Agents — The Behavioral Layer

An agent is not a separate program — it is a set of behavioral rules that define how the AI approaches work. Agents answer: What should I read first? When should I test? What should I update? How should I report results?

In Claude Code, agent behavior is defined in CLAUDE.md. In Codex and Antigravity, it goes in agents.md. The AI reads these instructions at the start of every conversation and follows them throughout.

Claude Code Subagents: Claude Code can spawn focused subagents — lightweight agents that handle a specific subtask in parallel. Think of them as specialists you delegate to. The main agent orchestrates, subagents execute.
Agent Example 1: Research Agent

Use this when you need the AI to explore your codebase before implementing anything.

Before implementing anything, research the codebase:

1. Find all existing API endpoints and list them
2. Identify the authentication middleware and how it works
3. Check the database schema and list all models
4. Review the test setup and identify the testing framework
5. Check existing route patterns and naming conventions

Report your findings in a structured format.
Do NOT make any changes — this is research only.
Agent Example 2: Implementation Agent

Use this when the AI should implement a feature following strict rules.

Implement the user profile feature following these rules:

1. Read Docs/sessions/session-06.md for exact requirements
2. Follow the existing route pattern in src/routes/
3. Add Pydantic schema validation for all inputs
4. Add unit tests in tests/ following existing test patterns
5. Run all tests after implementation
6. Update Docs/api-routes.md with new endpoints
7. Update Docs/progress.md marking session-06 tasks complete
8. Report: files changed, tests run, results, any blockers
Agent Example 3: Testing Agent

Use this to delegate a thorough testing sweep.

Run a complete test sweep on the current codebase:

1. Run all backend tests: pytest -v
2. Run all frontend tests: npm test
3. Run the linter: ruff check . && npm run lint
4. Check for TypeScript errors: npx tsc --noEmit
5. Start the app and verify it runs without errors

For each failure:
- Identify the root cause
- Classify as: bug, missing test, flaky test, or config issue
- Suggest a fix but do NOT apply it yet

Report all results in a structured table.

6.2 Skills — Reusable Procedural Knowledge

Skills are markdown files that teach the AI how to perform specific tasks. They live in .claude/commands/ and require no server, no API, no dependencies — just plain text instructions.

Think of it this way: MCP gives the AI access to tools (like a store's inventory). Skills tell the AI how to use those tools effectively (like having a knowledgeable employee). All the data in the world won't help without knowing what to do with it.

How to create a skill: Create a markdown file in .claude/commands/. The filename becomes the command name. Use $ARGUMENTS for dynamic input. Run with /project:command-name.
Skill 1: Run Session
.claude/commands/run-session.md
Read Docs/sessions/session-$ARGUMENTS.md

Complete only this session following CLAUDE.md rules.
Make small, atomic, testable changes.

For each task in the session:
1. Implement the change
2. Run relevant tests immediately
3. Fix any failures before moving to the next task

After all tasks are complete:
- Update Docs/progress.md (mark completed tasks)
- Update Docs/sessions/session-$ARGUMENTS.md (add implementation notes)
- Update Docs/api-routes.md (if any routes changed)
- Update Docs/frontend-pages.md (if any UI changed)
- Update Docs/readme.md (if setup steps changed)

Report: files changed, tests run, errors found, fixes applied, blockers.

Usage: /project:run-session 03

Skill 2: Add API Route
.claude/commands/add-api-route.md
Add a new API route: $ARGUMENTS

Steps:
1. Read Docs/api-routes.md for existing route patterns
2. Read the existing route files to match conventions
3. Create the route with proper HTTP method and path
4. Add Pydantic/Zod schema validation for request body
5. Add controller logic with proper error handling
6. Add unit tests following existing test patterns
7. Run tests: pytest -v (or npm test)
8. Update Docs/api-routes.md with the new endpoint
9. Update Docs/progress.md

Follow existing naming conventions exactly.
Do not change any existing routes.

Usage: /project:add-api-route "POST /api/users/:id/avatar"

Skill 3: Create Component
.claude/commands/create-component.md
Create a new frontend component: $ARGUMENTS

Steps:
1. Read Docs/frontend-pages.md for existing component patterns
2. Check existing components for naming and style conventions
3. Create the component file with TypeScript types
4. Add props interface with proper typing
5. Include loading, error, and empty states
6. Add responsive styles following existing patterns
7. Add unit test for the component
8. Run tests: npm test
9. Update Docs/frontend-pages.md

If the component needs API data, use the existing
data fetching pattern (React Query, SWR, etc.).

Usage: /project:create-component "UserProfileCard"

Skill 4: Review Code
.claude/commands/review-code.md
Review all changes made in the current session.

Check for:
1. Security: SQL injection, XSS, exposed secrets, auth bypass
2. Performance: N+1 queries, missing indexes, unnecessary re-renders
3. Error handling: unhandled promises, missing try/catch, silent failures
4. Type safety: any types, missing null checks, unsafe casts
5. Test coverage: untested happy paths, missing edge case tests
6. Convention compliance: naming, file structure, import order

For each issue found:
- File and line number
- Severity: critical / warning / suggestion
- What's wrong and why
- Suggested fix

Do NOT auto-fix. Report only.

Usage: /project:review-code

Skill 5: Run Tests
.claude/commands/run-tests.md
Run the full test suite and report results.

1. Run backend tests: pytest -v --tb=short
2. Run frontend tests: npm test -- --coverage
3. Run linter: ruff check . && npm run lint
4. Run type checker: npx tsc --noEmit

Report format:
## Test Results
- Backend: X passed, Y failed
- Frontend: X passed, Y failed
- Lint: clean / N issues
- Types: clean / N errors

## Failures (if any)
For each failure:
- Test name and file
- Error message
- Root cause analysis
- Suggested fix

## Coverage
- Overall: X%
- Untested areas: [list files with 0% coverage]

Usage: /project:run-tests

Skill 6: Setup Database
.claude/commands/setup-database.md
Set up or reset the development database.

Steps:
1. Read Docs/prd.md for data model requirements
2. Read existing migration files (if any)
3. Create/update database models matching the PRD
4. Generate migration files
5. Run migrations
6. Create seed data for development
7. Verify: connect and query each table
8. Update Docs/architecture.md with current schema

Database: use SQLite for local dev unless CLAUDE.md
specifies otherwise.

If the database already exists, ask before dropping.
Never drop a production database.

Usage: /project:setup-database

6.3 MCP Servers — Connecting AI to External Tools

MCP (Model Context Protocol) is a standardized way to connect AI agents to external tools and data sources. An MCP server is a standalone program that exposes tools, resources, and prompts over a standard protocol. The AI agent calls these tools like functions.

Important: Only add MCP servers you actually need. Each server adds complexity and potential failure points. Document what each server does, when to use it, and when NOT to use it.

MCP Server 1: Filesystem
.claude/settings.json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/project"]
    }
  }
}

Purpose: Read and write project files. Use when: editing code, docs, configs. Skip when: Claude Code already has built-in file access — you rarely need this one separately.

MCP Server 2: GitHub
.claude/settings.json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "ghp_your_token_here"
      }
    }
  }
}

Purpose: Create PRs, review code, manage issues, read repo metadata. Use when: syncing with tracked issues, creating PRs after sessions, reviewing code changes. Skip when: working entirely offline or on a local-only project.

MCP Server 3: PostgreSQL
.claude/settings.json
{
  "mcpServers": {
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "postgresql://user:pass@localhost:5432/mydb"
      }
    }
  }
}

Purpose: Query database, inspect schema, run migrations. Use when: building data models, debugging queries, verifying data. Skip when: using SQLite (just query via CLI) or no database in the project.

MCP Server 4: Playwright (Browser Automation)
.claude/settings.json
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["-y", "@anthropic-ai/mcp-playwright"]
    }
  }
}

Purpose: Navigate pages, click elements, fill forms, take screenshots, run UI tests. Use when: testing frontend features, verifying UI changes, debugging visual issues. Skip when: backend-only changes or API-only projects.

MCP Server 5: Stitch (Design-to-Code)
.claude/settings.json
{
  "mcpServers": {
    "stitch": {
      "command": "npx",
      "args": ["-y", "stitch-mcp"],
      "env": {
        "GOOGLE_CLOUD_PROJECT": "your-project-id"
      }
    }
  }
}

Purpose: Pull designs from Google Stitch, extract design tokens, fetch screen code and images. Use when: implementing UI from Stitch designs, maintaining design consistency. Skip when: no design system or using Figma instead. (See Module 08 for full Stitch setup.)

MCP Server 8: Memory
.claude/settings.json
{
  "mcpServers": {
    "memory": {
      "command": "npx",

Purpose: Persistent knowledge graph across sessions. Use when: you need the AI to remember decisions, preferences, and project context across conversations. Skip when: using Claude Code (it has built-in memory already).

MCP Boundary Rule: Always document which MCP does what and when NOT to use it. Too many unbounded MCP servers cause the AI to use the wrong tool. Add an # MCP Tools section to your CLAUDE.md listing each server's purpose and boundaries.

6.4 Tools — The Agent's Hands

Tools are the specific capabilities available to the agent. Each AI coding tool provides a different set:

Tool What It Does Available In
TerminalRun shell commands, scripts, testsAll tools
File EditorRead, create, edit, delete filesAll tools
BrowserNavigate pages, take screenshots, interact with UIClaude Code (via MCP), Antigravity
Database ClientQuery, inspect schema, run migrationsClaude Code (via MCP), Antigravity
GitCommit, branch, push, create PRsClaude Code, Codex
Test RunnerExecute test suites, report resultsAll tools (via terminal)

Document your available tools, standard commands, and expected output format in CLAUDE.md or agents.md.

6.5 Memory — The Persistent Brain

Memory is the persistent context layer that survives across sessions and conversations.

Good Memory Items
  • "Use React Query for server state"
  • "All routes must have schema validation"
  • "All sessions must update progress.md"
  • "Use SQLite during local dev"
  • "Deployment target is Vercel"
  • "User prefers concise responses"
Bad Memory Items
  • Temporary debugging notes
  • Random one-time errors
  • Unstable unapproved ideas
  • Ephemeral task details
  • Code snippets (use the repo instead)
  • Things already in CLAUDE.md
Claude Code Memory: Claude Code has a built-in memory system stored in ~/.claude/projects/<project>/memory/. Use it for user preferences, project decisions, and feedback. Memory persists across conversations automatically. You can also explicitly tell Claude Code: "Remember that we use Stripe for payments".
