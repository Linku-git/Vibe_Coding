Turn an idea into an implementation-ready PRD — including the right tech stack for your project.

What is prd.md?

The single source of truth for: project purpose, users, features, flows, constraints, technical assumptions, and success criteria — including your tech stack decision.

Stack selection is part of the PRD, not a separate step. Your tech stack shapes every session, every test, and every folder. Decide it here, document it here, and never revisit it mid-project without a strong reason.

Stack Decision Framework
1
Define project type and scale

SaaS, e-commerce, content site, API, or internal tool? Expected users: 10, 1K, or 100K?

2
Identify real-time requirements

WebSockets, live updates, or notifications? If yes, consider Node.js + Redis + Socket.io.

3
Evaluate team familiarity

Use what you know. A Python team should use FastAPI or Django, not learn Go for a CRUD app.

4
Check ecosystem and library availability

Payments, email, auth, file uploads — make sure libraries exist before committing.

5
Consider deployment target

Vercel for Next.js, Railway/Fly.io for Docker, AWS/GCP for enterprise.

6
Document in the PRD

Write the chosen stack, reasoning, and alternatives into the Tech Stack section of prd.md below.

Key Lesson

Pick the stack that matches your project's real constraints — team skills, scale requirements, deployment target — not the one with the most hype.

Step 2: Write the PRD

With your stack chosen, generate the full PRD. The Tech Stack section is where your decision from Step 1 gets documented permanently.

PRD Template

Docs/prd.md
# Project Name

## Overview
What the project is and why it exists.

## Goals
What success means.

## Target Users
Who uses it.

## Core Features
Main features with descriptions.

## User Flows
How users move through the app.

## Functional Requirements
Exact system behavior.

## Non-Functional Requirements
Performance, security, responsiveness, accessibility.

## Tech Stack
Frontend, backend, DB, auth, deployment.

## Pages / Screens
All UI pages with purpose.

## API / Backend Modules
Routes, controllers, services.

## Data Models
Main entities and relationships.

## Risks / Edge Cases
Known issues or complexity.

## Milestones
High-level implementation roadmap.

## Acceptance Criteria
What must be true for completion.
